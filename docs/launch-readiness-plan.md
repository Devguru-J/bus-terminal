# BusTerminal Launch Readiness Plan

Claude Code 작업용 출시 전 개선 명세서입니다.

이 문서는 BusTerminal을 본격 출시하기 전에 초보 개발자와 익숙한 개발자 모두가 덜 헷갈리게 만들기 위한 작업 기준입니다. 핵심 목표는 기능을 크게 늘리는 것이 아니라, 첫 방문자가 무엇을 해야 하는지 바로 이해하게 만드는 것입니다.

## 목표

- 초보 개발자가 첫 방문 후 3분 안에 첫 설정 파일을 만들 수 있게 한다.
- 기존 개발자는 테마, 폰트, export, route 저장 흐름을 빠르게 파악할 수 있게 한다.
- BusTerminal의 전광판, 환승센터, 승강장 컨셉은 유지한다.
- 전체 리디자인이 아니라 출시 전 사용성 정리와 문구 개선에 집중한다.
- README, Guide, 실제 UI의 설명이 서로 모순되지 않게 맞춘다.

## 절대 유지할 것

- 전광판 / 네온 사인 / 터미널 환승센터 컨셉
- 어두운 터미널 배경과 초록 포인트 컬러
- BusTerminal, 승강장, 환승센터, 노선 같은 브랜드 메타포
- 현재의 sidebar + topbar + platform page 구조
- 기존 라우팅과 store 구조
- 기존 기능의 동작 방식

## 하지 말 것

- 전광판 디자인을 제거하지 말 것.
- 환승센터 컨셉을 일반 SaaS 대시보드처럼 바꾸지 말 것.
- 전체 레이아웃을 갈아엎지 말 것.
- 불필요한 랜딩 페이지식 섹션을 추가하지 말 것.
- 카드와 장식 요소를 과하게 늘리지 말 것.
- 새 라이브러리를 추가하지 말 것. 꼭 필요하면 먼저 이유를 명확히 남길 것.
- 기능 구현 범위를 벗어난 대규모 리팩터링을 하지 말 것.

## 현재 상태 요약

BusTerminal은 이미 출시형 구조에 가까워져 있다.

- `/guide` 사용 안내 페이지가 있다.
- 첫 방문 온보딩 모달이 있다.
- platform별 PageGuideCard가 있다.
- Privacy, Terms, 404, lazy route, ErrorBoundary, analytics, local backup, cloud snapshot 구조가 있다.
- `npm run build` 통과.
- `npm run test` 통과. 확인 당시 8개 파일 / 70개 테스트 통과.

남은 핵심 문제는 기능 부족보다 첫 사용 흐름과 문구 명확성이다.

## P0: 출시 전 반드시 수정

> **이번 작업 범위 가이드**
> - 즉시 실행 가능: #1 README Import / #2 Export 빈 상태 / #3 모바일 TopBar / #4 Home CTA — 4건 모두 사양 명확, 회귀 위험 낮음
> - 사양 보강 후 실행: #5 버튼 보조 설명 / #6 플랫폼 상단 문구 — 영향 파일 많고 카피 톤 합의 필요. 시간 부족 시 P1로 강등 권장

### 1. README와 실제 UI의 Import 상태 불일치 수정

파일: `README.md`

README에는 Import가 Planned로 적혀 있지만 실제 UI에는 여러 `환승하기` 기능이 존재한다.

수정 방향:

- 실제 구현된 import 범위를 기준으로 README의 Supported Platforms 표를 고친다.
- 아직 완전하지 않은 import라면 `Partial` 또는 `Experimental`로 표기한다.
- "Import는 Planned"라는 문장은 현재 구현 상태에 맞게 수정한다.

예시:

```md
> Import는 플랫폼별로 지원 범위가 다릅니다. 현재 일부 설정 파일은 Experimental import로 제공되며, 인식하지 못한 줄은 사용자에게 표시됩니다.
```

### 2. Export 빈 상태 문구 모순 수정

파일: `src/pages/Export.tsx`

현재 빈 상태에서는 "아직 다운로드할 설정이 없어요"라고 하면서, 아래 진단 영역에는 "점검 결과 문제가 없습니다. 바로 출발해도 됩니다."가 표시된다.

이 문구는 초보자에게 모순처럼 보인다.

수정 방향:

- 아무 설정도 수정되지 않은 상태에서는 진단 영역을 숨기거나 별도 빈 상태 문구를 보여준다.
- 추천 문구:
  - "아직 점검할 설정이 없습니다. 먼저 승강장에서 설정을 바꿔 주세요."
  - "설정을 변경하면 이곳에서 충돌과 누락을 점검합니다."

### 3. 모바일 topbar 중복 홈 아이콘 숨김

파일: `src/components/shell/TopBar.tsx`

모바일에서 topbar에 메뉴, 로고, 홈, 출발 전 점검, 설정, 계정 아이콘이 모두 표시된다.

문제:

- 로고 자체가 이미 홈 링크이므로 홈 아이콘은 모바일에서 중복이다.
- 390px 폭 기준으로 상단 아이콘이 답답해 보인다.

수정 방향:

- 홈 아이콘 링크에 `hidden lg:inline-flex` 또는 동등한 반응형 클래스를 적용한다.
- `출발 전 점검` 아이콘은 핵심 동선이므로 유지한다.
- 데스크톱에서는 현재 구성을 유지한다.

### 4. Home CTA 정합성 수정

파일: `src/pages/Home.tsx`

현재 `시작하기` 버튼이 바로 `/ghostty`로 이동한다.

수정 방향:

- 링크를 `/ghostty`로 유지한다면 버튼 문구를 더 정직하게 바꾼다.
- 추천 문구: `Ghostty로 시작하기`
- 네온사인 보드 아래 CTA 위치와 스타일은 유지한다.

주의:

- `내 도구 선택하기`처럼 포괄적인 문구를 쓰려면 링크도 `/guide` 또는 도구 선택 영역으로 바꿔야 한다.
- 이번 출시 전 작업에서는 회귀 위험이 가장 낮은 `Ghostty로 시작하기`를 우선한다.

문구 방향:

- `환승하기`

  - 추천 title: "기존 설정 파일을 가져와 현재 화면에 적용"

실행 전 결정해야 할 것:

- 병기 포맷: 괄호(`환승하기 (기존 설정 가져오기)`) / em dash / 별도 subtitle 중 어느 패턴?

- 반복 위치의 title attribute 통일안: 모든 페이지 동일 문구 vs 페이지별 변형 허용?

### 6. 플랫폼 상단 설명 문구 자연스럽게 수정

예: `src/pages/Ghostty.tsx`

가장 사람이 가까운 터미널. 폰트·색·창 패딩을 슬라이더로 잡고, 우측 미리보기에서 즉시 확인. 완성하면 ghostty config 한 줄로 출발합니다.

  - `src/pages/Zsh.tsx`
  - `src/pages/Helix.tsx`

  - `src/pages/Iterm2.tsx`

문구 원칙:

- 초보자가 이해할 수 있는 한국어를 우선한다.
- 도구명과 파일명은 정확히 쓴다.
- "출발", "환승" 같은 브랜드 언어는 유지하되 기능 설명과 같이 쓴다.
- 너무 감성적인 문장보다 실제 행동을 알려주는 문장을 우선한다.

## P1: 출시 전 사용성 개선

### 1. Guide 페이지를 "문서"보다 "행동 가이드"로 강화

파일: `src/pages/Guide.tsx`

현재 Guide는 설명이 잘 되어 있지만 문서형에 가깝다.

수정 방향:

- 첫 화면에서 바로 3가지 시작 경로를 보여준다.
- 추천 경로:
  - "처음이라 막막해요" -> Ghostty -> 테마 센터 -> 폰트 센터 -> Export
  - "기존 설정이 있어요" -> 해당 승강장 -> 환승하기
  - "테마와 폰트만 고르고 싶어요" -> 테마 센터 -> 폰트 센터
- 용어집은 아래로 내리고, 행동 경로를 위로 올린다.
- 버튼은 실제 이동 가능한 링크로 연결한다.

### 2. Export 이후 적용 안내 추가

파일: `src/pages/Export.tsx`

다운로드 이후 초보자는 "이 파일을 어디에 넣지?"에서 막힌다.

추가 방향:

- 다운로드 대상별로 target path를 이미 가지고 있으므로 이를 활용한다.
- 다운로드 후 또는 파일 목록 영역에 다음 정보를 보여준다.
  - 파일명
  - 적용 위치
  - 기존 파일 백업 권장
  - 설치 스크립트는 고급 사용자용이라는 경고

예시 문구:

```txt
다운로드한 파일은 바로 적용되지 않습니다. 기존 설정을 백업한 뒤 각 도구의 설정 경로에 직접 옮겨 주세요.
```

### 3. Theme 적용 행동의 안전성 안내

파일: `src/pages/Themes.tsx`

테마 센터에서 "모든 도구에 적용"은 강력한 기능이다.

현재 undo toast가 있다면 좋다. 다만 초보자에게는 "여러 도구 설정이 바뀐다"는 점이 더 명확해야 한다.

수정 방향:

- 적용 dock 또는 버튼 근처에 짧은 설명 추가:
  - "선택한 도구의 색상 설정만 바뀝니다."
  - "적용 후 되돌리기 버튼이 잠시 표시됩니다."

### 4. 온보딩 모달 모바일 UX 점검

파일: `src/components/shell/OnboardingModal.tsx`

첫 방문 온보딩 모달은 초보자 진입에 중요하다.

수정 방향:

- 모바일 390px 폭에서 버튼, 본문, 단계 표시가 잘리지 않는지 확인한다.
- 모달 자체가 첫 화면의 전광판 컨셉을 가리지 않도록 문구를 짧게 유지한다.
- 닫기, 건너뛰기, 다음 버튼이 엄지 조작에 충분한 크기인지 확인한다.

## P2: 출시 후 점진 개선

### 1. 브라우저 기본 prompt / confirm 개선

관련 파일 예시:

- `src/pages/Ghostty.tsx`
- `src/pages/Tmux.tsx`
- `src/pages/Neovim.tsx`
- `src/pages/Zsh.tsx`
- `src/pages/Helix.tsx`
- `src/pages/Warp.tsx`
- `src/pages/Iterm2.tsx`
- `src/pages/Settings.tsx`
- `src/pages/Profile.tsx`

현재 여러 곳에서 `window.prompt`, `window.confirm`을 사용한다. 출시 제품 느낌을 위해 앱 스타일 모달로 바꾸는 것이 좋다.

단, 확인된 인스턴스가 많으므로 출시 직전 한 번에 모두 바꾸지 않는다. 회귀 위험을 줄이기 위해 출시 후 1주 안에 점진적으로 처리한다.

우선순위:

1. 저장 이름 입력 모달
2. 위험 작업 확인 모달
3. keybind / plugin / workflow 추가 모달

주의:

- 한 번에 전부 바꾸기 어렵다면 저장 이름 입력과 위험 확인부터 바꾼다.
- 기존 기능을 깨지 말 것.
- 모달 디자인은 기존 `Modal`, `Button`, `Field` 컴포넌트 패턴을 따른다.
- 모달을 추가할 때 focus trap, Esc 닫기, 기존 Modal 접근성 패턴을 유지한다.
- Zustand 구독 패턴을 새로 추가할 때 autosave/hydration 회귀가 생기지 않게 주의한다.

### 2. Plausible analytics 활성화 체크리스트

파일:

- `src/lib/analytics.ts`
- `src/pages/Privacy.tsx`

현재 `VITE_PLAUSIBLE_DOMAIN` 환경변수가 비어 있으면 analytics는 no-op이고 외부 요청을 보내지 않는다. 출시 시점에 env var를 켜지 않는다면 별도 동의 UX는 필요 없다.

Plausible을 실제로 켤 때 확인할 것:

- Privacy 페이지에 analytics가 활성화 중임을 명확히 표시한다.
- 쿠키 없음, IP 익명화, self-host 여부를 실제 설정 기준으로 확인한다.
- 필요하면 사용자가 analytics를 거부할 수 있는 opt-out 안내를 추가한다.

많은 사람이 쓰게 하려면 빈 화면에서 직접 조립하게 하는 것보다 바로 눌러볼 수 있는 프리셋이 효과적이다.

- 회사 노트북용

### 4. 공유 기능 강화

수익화보다 확산이 목적이면 공유 기능이 중요하다.

추천:

- 브랜드 용어는 기능 설명과 함께 쓴다.

예:

```txt
환승하기
기존 설정 파일을 가져와 현재 화면에 적용합니다.
```

```txt
출발권 만들기
완성한 설정을 파일로 다운로드합니다.
```

```txt
차고 보관
현재 설정 조합을 내 노선에 저장합니다.
```

### 피할 표현

- 너무 감성적이라 기능이 흐려지는 문장
- "한 줄로 출발합니다"처럼 실제 동작과 다르게 해석될 수 있는 문장
- 영어와 한국어가 불필요하게 섞인 문장
- 같은 의미의 브랜드 용어를 한 화면에 너무 많이 쓰는 것

## 추천 작업 순서

1. README와 실제 UI 설명 불일치 수정
2. Export 빈 상태 문구 모순 수정
3. 모바일 topbar 홈 아이콘 숨김
4. Home CTA 문구를 `Ghostty로 시작하기`로 정합성 수정
5. 플랫폼 페이지 상단 subtitle과 PageGuideCard 문구 개선
6. Guide / Onboarding 문구와 모바일 UX 점검
7. prompt / confirm 모달화는 출시 후 점진 작업으로 분리
8. build/test 실행

## 검증 기준

작업 후 반드시 확인한다.

```bash
npm run build
npm run test
```

가능하면 브라우저에서 아래 화면을 확인한다.

- `/`
- `/guide`
- `/ghostty`
- `/themes`
- `/export`
- 모바일 390px 폭의 `/`와 `/ghostty`

확인할 것:

- 전광판 디자인이 유지되는가
- 첫 방문자가 다음 행동을 알 수 있는가
- Export 빈 상태가 모순되지 않는가
- README와 UI 기능 설명이 일치하는가
- 모바일에서 상단 아이콘이 답답하지 않은가
- 기존 테스트가 통과하는가

## Claude Code 실행 프롬프트 예시

```txt
docs/launch-readiness-plan.md 파일을 먼저 읽고, P0 #1~#4를 먼저 구현해줘. P0 #5, #6은 각 항목의 "실행 전 결정해야 할 것"이 합의되기 전까지는 손대지 마. 시간이 남으면 P1 항목까지 진행해줘. P2는 구현하지 말고 별도 TODO로만 남겨줘.

중요:
- 기존 전광판/환승센터/터미널 컨셉은 절대 제거하지 마.
- 전체 리디자인 금지.
- 디자인 톤은 유지하고, 초보자가 이해하기 어려운 문구와 첫 사용 흐름만 개선해줘.
- 새 라이브러리는 추가하지 마.
- README와 실제 UI의 기능 설명이 다르면 실제 구현 기준으로 문서를 맞춰줘.
- 이번 작업에서 window.prompt / window.confirm 전체 모달화는 하지 마. 출시 후 점진 작업으로 남겨줘.
- Home CTA는 링크가 /ghostty로 유지된다면 문구를 "Ghostty로 시작하기"로 맞춰줘.
- 모바일 topbar에서는 로고가 홈 링크이므로 홈 아이콘만 숨기고, 출발 전 점검 아이콘은 유지해줘.
- 작업 후 npm run build, npm run test를 실행하고 결과를 요약해줘.
```
